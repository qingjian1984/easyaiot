# 天地图 UI / 交互升级计划清单

> 范围：`WEB/src/components/TiandituMap` 及其在 `/alert`（告警一张图）、设备监控、轨迹回放、选址等业务页的应用。
> 原则：**默认不改动后端**；凡需后端配合的项已单独标注 `【需后端】`，并给出可先落地的纯前端降级方案。

---

## 一、调研结论 · 可借鉴的成熟方案要点

### 1. 天地图平台能力（用足官方服务）
- **多底图类型**：矢量 `vec` / 影像 `img`（原生 2–18 级）/ 地形晕渲 `ter`（仅 1–14 级），各带独立中文注记 `cva/cia`。本项目目前只用了 `vec/img`。
- **WMTS 原生最高 18 级**：超过 18 级官方无瓦片。成熟做法是 `maxNativeZoom=18` + `maxZoom` 放更高，让客户端**插值放大**而不是直接封顶或变空白。
- **Web 数据服务**：POI 搜索、地理/逆地理编码、行政区划边界、路径规划。本项目已接入 POI 搜索与逆地理，但只用在"选址"，未用于一张图的"搜索定位/区划筛选"。
- **配额限制**：Web 接口有频率限制（超限返回 418/非 JSON），需缓存 + 容错（本轮已加容错）。

来源：[天地图 API 服务](http://lbs.tianditu.gov.cn/server/MapService.html)、[天地图 WMTS 层级](https://blog.csdn.net/sinat_31213021/article/details/126589999)、[奥维-天地图底图最大级别](https://www.ovital.com/131425-2/)、[2025 中国地图 API 评测](https://www.explinks.com/blog/pr-2025-china-map-api-comprehensive-evaluation-and-developer-guide/)

### 2. 安防视频监控"一张图"交互范式
- **专用监控图层 / 底图分离**：监控点位独立成层，与底图解耦，图层可独立开关。
- **点聚合解决撒点重叠**：海量摄像头平面撒点会局部重叠、无法精选——高层级聚合、低层级展开是标配（本项目已用 Cluster）。
- **图标区分设备类型与状态**：枪机/球机/卡口用不同图标，在线/离线/告警用不同颜色。
- **点击图标即调实时视频、悬停看属性**。
- **空间查询**：点/线/面周边查询、框选、不规则选择。
- **状态可视化 + 多源叠加**：设备状态、告警、警力、应急等同图叠加。
- **电子围栏 + 入侵联动**、**视域分析（相机可视区域多边形）**、**轨迹回放**、**热力图（密度）** 是进阶能力。

来源：[GIS+安防一张图功能](https://www.jimay.com/support/1371.html)、[GIS 与视频监控集成](https://www.cnblogs.com/d1012181765/p/13804550.html)、[GIS 在安防的行业应用](https://www.jimay.com/technews/38388.html)、[EasyCVR 设备上图](https://cloud.tencent.com/developer/article/2064009)

### 3. OpenLayers 渲染性能
- 几千点内：普通 `VectorLayer` + 可视范围过滤即可。
- 上万点：`Cluster` 聚合 / `WebGLVectorLayer`（GPU，支持十万级）。
- 仅看密度：`ol/layer/Heatmap`。
- 当前摄像头量级约 1000、告警 `pageSize=500`，**Cluster 足够**；为未来万级预留 WebGL 切换即可。

来源：[OpenLayers 海量点 WebGL](https://blog.csdn.net/HerryDong/article/details/136923391)、[OpenLayers 点聚合](https://www.cnblogs.com/jiujiubashiyi/p/16557061.html)、[ol Heatmap API](https://openlayers.org/en/latest/apidoc/module-ol_layer_Heatmap-Heatmap.html)

## 二、现状对照（已具备 vs 主要差距）

| 能力 | 现状 | 差距 |

| 底图切换 | vec / img | 缺地形 ter、注记独立开关 |

## 六、结合 GB28181 数据的进阶优化（调研结论）

### 数据可达性（成本前提）
地图数据来自 VIDEO `/video/camera/locations`（`list_devices_for_map`）。现状：
- VIDEO 只落库 **经纬度/地址 + 通用 `support_move`/`support_zoom`**。
- 所有 GB28181 富属性（`ptzType`/`directionType`/`positionType`/`roomType`/`useType`/`supplyLightType`/`resolution`）**只在 DEVICE/iot-gb28181**。
- **关键利好**：VIDEO 的 `gb28181_sync_service` 已在从 WVP HTTP 拉取整段通道 JSON（其中就含这些字段），目前只取了坐标——所以"抓取落库"是**扩展现有同步 + 加列**，不是新建管道。
- 实时 `MobilePosition` / 轨迹：VIDEO 有表但**无写入代码**（轨迹回放目前在读空表）→ 需**新建采集管道**（成本高）。
- GB28181 设备报警 `DeviceAlarm`：独立源，WEB 有 `queryDeviceAlarm` 绑定但未使用；地图告警目前**只来自 AI 算法**（`queryAlarmList`）。
- `directionType` 是 **8 方位离散枚举**（光轴方位），固定相机无连续航向；连续航向只有移动设备 `MobilePosition.direction`。

### 优化机会清单

| # | 借助的 GB28181 数据 | 地图收益 | 后端成本 | 优先级 |
|---|---|---|---|---|
| G-1 | **PTZType** | 精确结构图标(球机/半球/固定枪机/遥控/多目)，替代 support_move/zoom 推断 | 扩展现有 VIDEO 同步+加列 | 高 |
| G-2 | **DirectionType** | 固定相机 8 方位**视域扇形**（"看哪"终于数据驱动） | 同 G-1（同一次同步扩展） | 高 |
| G-3 | **PositionType / UseType** | 按业务分类(卡口/党政/校园/治安复杂；治安/交通/重点)**筛选/着色/图例** | 同 G-1 | 中高 |
| G-4 | **RoomType / SupplyLightType / Resolution** | 弹窗详情 + 场景筛选(夜视/室内外/分辨率) | 同 G-1 | 中 |
| G-5 | **PTZ 控制接口(已存在)** | 地图选中球机→内联预览旁加**云台方向/变倍**控制 | **纯前端**（VIDEO `/device/<id>/ptz` 已有；GB28181 经 WVP） | 高(性价比最优) |
| G-6 | 通道 online(已有) | "只看离线/掉线闪烁"快速发现故障设备 | 纯前端 | 中 |
| G-7 | **MobilePosition(实时)** | 车载/单兵/无人机**实时移动目标上图**(连续航向+速度) | **高**：新建采集管道(订阅/轮询 WVP→落库/推流) | 中(招牌能力，按需立项) |
| G-8 | MobilePosition(历史) | **激活已有但空转的轨迹回放**(把历史写入 VIDEO 轨迹表) | 高(历史采集) | 中 |
| G-9 | **DeviceAlarm** | 设备级报警(视频丢失/移动侦测/故障/SOS)上图，与 AI 告警分层 | 中(WEB 绑定已存在，需接入+VIDEO 聚合) | 中 |

### 建议落地顺序
1. **G-5（纯前端，最高性价比）**：复用已建的 P1-4 内联预览 + 现有 PTZ 接口，球机直接在图上转动/变倍。
2. **G-1~G-4 一批**：一次 VIDEO `gb28181_sync_service` 扩展（多取几个字段 + Device 加列 + locations 响应透出）即可解锁"精确结构图标 + 视域扇形 + 业务分类筛选 + 详情/场景筛选"。前端按字段渲染。
3. **G-7/G-8（实时与轨迹）**：若有车载/单兵/无人机设备，作为 GB28181 招牌能力立项——需新建 MobilePosition 采集管道（VIDEO 当前完全缺失）。
4. **G-9 设备报警**：按运维需要接入。

> 注：G-1~G-4、G-7~G-9 的后端改动都在 VIDEO（及其对 WVP 的同步），按项目记忆需先同步到服务器副本再构建/重启。
